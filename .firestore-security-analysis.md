# Firestore security analysis

Firebase Authentication protects this browser inventory app. `products`,
`suppliers`, `warehouses`, and `purchaseOrders` are scoped by immutable `ownerId`. Queries
filter on the signed-in UID before sorting. Supplier contact details are
owner-only. Creates and updates use strict schemas; all unknown paths are denied.

Only the Firebase account whose token email is `chrisdiorsystem@gmail.com` can
access business collections. Other authenticated accounts run the client-side
read-only demo and are denied all Firestore reads and writes.

## Red-team pass

- Public list/read: denied because every product read requires matching owner ID.
- Viewer bypass: denied server-side because ownership also requires the configured administrator email.
- Cross-user CRUD: denied by `isDocOwner`, `uidUnchanged`, and `uidNotModified`.
- Update/schema bypass: denied because `isValidProduct` runs on create and update.
- Ownership/created timestamp hijacking: denied; both fields are immutable.
- Type juggling, oversized strings, negative/huge values: denied by type/range/size checks.
- Missing/extra fields: denied by `hasAll` and `hasOnly`.
- Timestamp manipulation: denied by the recent server timestamp check.
- Query mismatch: the client filters `ownerId == request.auth.uid` before ordering.
- Unknown/orphaned paths and privilege fields: denied by the recursive default rule.
- Sales orders: owner-only CRUD, strict schema, sequential status transitions only.
- Purchase orders: owner-only create/read/delete, 1-10 deeply validated line items.
- Suppliers: owner-only CRUD with strict contact, aggregate, status, and timestamp validation.
- Warehouses: owner-only read/create/update with strict identity, capacity, status, and timestamp validation.
- Warehouse deletion is denied so products and orders cannot be orphaned by a client-side delete.
- Warehouse query: `ownerId == uid`, ordered by `name`; it cannot return another owner's locations.
- Customers: owner-only CRUD with strict PII, aggregate, status, and timestamp validation.
- Stock adjustments: owner-only read/create; immutable audit records with bounded quantities.
- User invitations: owner-only read/create/delete; restricted to non-admin roles and Pending status.

The invitation model deliberately cannot create Firebase Authentication users,
grant roles, or assign admin privileges. Account provisioning remains an
out-of-band administrator action.
